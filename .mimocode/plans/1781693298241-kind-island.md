# Podium Windows 迁移计划：KMP + Compose Multiplatform

## 项目概览

将现有的 Android 播客应用 Podium 迁移到 Windows 桌面平台，使用 KMP + Compose Multiplatform 方案。

**关键策略**：
- **开发期间**：Android 代码和 KMP 代码共存，互不干扰
- **迁移完成后**：删除所有 Android 特定代码（`app/` 模块、Android 构建配置等）
- **最终状态**：项目仅包含 Windows 桌面应用代码

## 技术选型确认

| 层 | 方案 |
|---|---|
| 桌面平台 | 仅 Windows |
| 媒体播放 | vlcj (libvlc JVM 绑定) |
| 数据库 | SQLDelight |
| 导航 | Compose Multiplatform Navigation |
| 项目结构 | Feature 模块化 |

## 渐进式迁移策略

### 第一阶段：项目骨架搭建

**目标**：建立 KMP 多模块结构，保持现有 `app/` 模块不变。

**步骤**：

1. **修改 `settings.gradle.kts`**
   - 保留 `include(":app")`
   - 添加新模块: `shared`, `feature-home`, `feature-discover`, `feature-player`, `feature-library`, `feature-settings`, `desktop`
   
2. **创建根 `build.gradle.kts` 插件配置**
   - 添加 Kotlin Multiplatform 插件
   - 添加 Compose Multiplatform 插件
   - 添加 SQLDelight 插件
   - 更新 version catalog (`libs.versions.toml`)

3. **创建 `shared` 模块** (`shared/build.gradle.kts`)
   - 定义 Kotlin Multiplatform target: `jvm` (Windows)
   - 定义共享 API 层
   - 依赖: Ktor, SQLDelight, kotlinx.serialization, kotlinx.coroutines

4. **创建 `desktop` 模块** (`desktop/build.gradle.kts`)
   - Kotlin JVM application 插件
   - Compose Desktop 配置
   - 主入口 `main()` 函数
   - 打包配置 (可选: jpackage)

### 第二阶段：共享数据层迁移

**目标**：将数据库模型和 DAO 迁移到 SQLDelight。

**步骤**：

1. **SQLDelight Schema 定义** (`shared/src/commonMain/sqldelight/`)
   - 创建 `app/podiumpodcasts/podium/sqldelight/` 目录
   - 定义 `.sq` 文件:
     - `Podcast.sq` - 播客表 CRUD
     - `PodcastEpisode.sq` - 单集表 CRUD
     - `PodcastEpisodePlayState.sq` - 播放状态表
     - `PodcastHistory.sq` - 历史记录表
     - `PodcastSubscription.sq` - 订阅表
     - `PodcastEpisodeDownload.sq` - 下载表
     - `List.sq` - 列表表
     - `ListItem.sq` - 列表项表
     - `SyncAction.sq` - 同步动作表
     - `UpdatePodcastRun.sq` - 统计表

2. **共享数据模型** (`shared/src/commonMain/kotlin/`)
   - 创建 `data/model/` 包
   - 定义平台无关的数据类 (去掉 Room 注解、Android 特定依赖)
   - 保持与现有模型字段一致

3. **共享 Repository 层**
   - 创建 `data/repository/` 包
   - 定义 `PodcastRepository`, `EpisodeRepository`, `SettingsRepository` 等接口
   - 实现跨平台的数据库操作

4. **迁移 SettingsRepository**
   - 使用 `multiplatform-settings` 库替代 Android DataStore
   - 保持相同的 API 签名

### 第三阶段：共享业务逻辑迁移

**目标**：将 Manager 层和 API 客户端迁移到共享模块。

**步骤**：

1. **共享 Manager 层**
   - `PodcastManager` - 播客管理逻辑
   - `SubscriptionManager` - 订阅管理
   - `SyncManager` - 同步管理
   - `ExportManager` - 导出管理
   - `DownloadManager` - 下载管理 (需适配桌面文件系统)

2. **共享 API 客户端**
   - `FetchPodcastClient` - RSS 获取 (使用 Ktor，已跨平台)
   - `ApplePodcastClient` - Apple Podcasts API
   - `SyncClient` - gpodder 同步客户端
   - RSS 解析 (使用 `rssparser`，支持多平台)

3. **共享工具类**
   - `sha256` 哈希工具
   - XML 解析工具
   - JSON 序列化配置

### 第四阶段：共享 UI 层迁移

**目标**：将 Composable 函数迁移到共享模块。

**步骤**：

1. **Theme 迁移**
   - 迁移 `Color.kt`, `Theme.kt`, `Type.kt`
   - 使用 Compose Multiplatform 的 Material 3

2. **组件迁移** (`shared/src/commonMain/kotlin/ui/component/`)
   - `MediaItemListItem` - 单集列表项
   - `FloatingMediaPlayer` - 浮动播放器
   - 通用按钮、对话框组件
   - 导航栏组件

3. **路由迁移** (`shared/src/commonMain/kotlin/ui/route/`)
   - Home, Discover, Library, Settings 路由
   - PodcastDetail, EpisodeDetail 路由
   - Downloads, History, Lists 路由

4. **ViewModel 迁移**
   - 使用 Compose Multiplatform 的 ViewModel
   - 迁移 `MainViewModel`, `MediaPlayerViewModel` 等
   - 使用平台无关的状态管理

### 第五阶段：Windows 桌面实现

**目标**：实现 Windows 特定的功能。

**步骤**：

1. **媒体播放器** (`desktop/src/main/kotlin/player/`)
   - 使用 vlcj 封装播放器
   - 实现 `MediaPlayer` 接口
   - 支持播放、暂停、快进、快退
   - 支持播放速度调节
   - 支持睡眠定时器

2. **下载管理** (`desktop/src/main/kotlin/download/`)
   - 实现桌面下载逻辑
   - 使用 Ktor 下载文件
   - 管理本地文件存储

3. **设置管理** (`desktop/src/main/kotlin/settings/`)
   - 使用 `multiplatform-settings` 桌面实现
   - 文件存储路径: 用户目录下的配置文件

4. **通知系统** (`desktop/src/main/kotlin/notification/`)
   - 桌面通知 (可选，Windows Toast 通知)

5. **系统集成**
   - 文件关联 (OPML 导入)
   - 桌面快捷键支持
   - 系统托盘集成 (可选)

### 第六阶段：主入口和打包

**目标**：完成桌面应用主入口和打包。

**步骤**：

1. **主入口** (`desktop/src/main/kotlin/main.kt`)
   - 初始化 Compose Desktop
   - 配置窗口参数
   - 设置数据库
   - 启动应用

2. **打包配置**
   - 配置 jpackage 或其他打包工具
   - 创建 Windows 安装程序 (MSI/EXE)
   - 配置应用图标和元数据

## 注意事项

1. **开发期间共存**：现有 Android 代码不做任何修改，KMP 代码独立开发
2. **迁移完成后删除**：当 KMP 版本功能完整后，删除以下内容：
   - `app/` 整个模块
   - `app/build.gradle.kts`
   - `app/src/` 目录
   - Android 特定的 Gradle 插件配置
   - Android 特定的版本目录条目
   - `gradlew` 中的 Android 构建脚本
3. **最终项目结构**：只保留 `shared/`, `desktop/`, `feature-*/` 模块和根构建配置
4. **渐进式迁移**：每个阶段独立可验证，避免大规模重构
5. **共享代码优先**：尽可能将逻辑放在 shared 模块，减少桌面特定代码
6. **测试覆盖**：每个阶段添加单元测试，确保功能正确
